## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Test Infrastructure] Missing afterEach cleanup in component tests**
- **Found during:** Task 2 (badge.test.tsx initial run)
- **Issue:** Test 3 ("does NOT render dismiss button") failed because DOM from Test 2's render leaked into Test 3's screen queries. Project has `@testing-library/react` + `@testing-library/jest-dom` installed but no Vitest setup file auto-importing RTL's afterEach cleanup.
- **Fix:** Added explicit `afterEach(() => cleanup())` at the top of each new component test file (matches pattern used in existing Phase 2 `tests/unit/company-logo.test.tsx` and `source-badge.test.tsx`).
- **Files modified:** `tests/unit/ui/badge.test.tsx`, `tests/unit/ui/slider.test.tsx`, `tests/unit/metrics/sparkline.test.tsx`, `tests/unit/metrics/delta.test.tsx`, `tests/unit/metrics/logo-tile.test.tsx`
- **Commits:** `dc6f60a`, `2665de1`, `b83b0ed`

**2. [Rule 1 — Test Accuracy] happy-dom preserves inline-style color as authored hex (not rgb)**
- **Found during:** Task 3 (delta.test.tsx + logo-tile.test.tsx initial run)
- **Issue:** Plan's acceptance criteria hard-coded `rgb(12, 138, 58)` but happy-dom returns `#0C8A3A` verbatim. jsdom normalizes to rgb(); happy-dom does not.
- **Fix:** Added a `colorMatches(actual, hex)` / `bgMatches(actual, hex)` helper in each test file that accepts either representation. Component behavior is correct — only the assertion needed to tolerate both forms.
- **Files modified:** `tests/unit/metrics/delta.test.tsx`, `tests/unit/metrics/logo-tile.test.tsx`
- **Commit:** `2665de1`

**3. [Rule 1 — Test Accuracy] Recharts ResponsiveContainer yields 0×0 in happy-dom**
- **Found during:** Task 3 (sparkline.test.tsx initial run)
- **Issue:** Plan's acceptance criterion `container.querySelector('svg')` returned null because Recharts' `ResponsiveContainer` depends on `ResizeObserver` + element layout, which happy-dom reports as 0. No SVG actually rendered.
- **Fix:** Pivoted test contract to assert on the ChartContainer wrapper (`[data-chart]` element) plus the inline `width`/`height` style propagation — these are the genuinely unit-testable invariants. SVG render is a visual/e2e concern.
- **File modified:** `tests/unit/metrics/sparkline.test.tsx`
- **Commit:** `2665de1`

**4. [Rule 2 — Plan Gap] Slider single-`<Thumb/>` did not support dual-thumb rendering**
- **Found during:** Task 1 (inspecting CLI-generated slider.tsx)
- **Issue:** shadcn CLI generated slider.tsx renders a single hard-coded `<SliderPrimitive.Thumb />`. Radix Slider supports N thumbs but requires the caller to render one Thumb per value-array entry. Without this fix the Wave 4 FacetRangeInputs `value={[min, max]}` call site would only get one thumb.
- **Fix:** Wrapped the Thumb render in `Array.from({ length: thumbCount }).map(...)` where `thumbCount` is derived from `(value ?? defaultValue).length` (defaults to 1 for single-value mode). This is an essential correctness fix for D-02.3 — dual-thumb range is the primary consumer.
- **File modified:** `src/components/ui/slider.tsx`
- **Commit:** `1e4b2fb`
- **Test coverage:** `tests/unit/ui/slider.test.tsx` (added in Task 4) locks the contract: `value={[10,50]}` → 2 thumbs, `value={[30]}` → 1 thumb.

**5. [Plan Authoring] shadcn CLI installed Recharts v2 despite RESEARCH forecast of v3**
- **Found during:** Task 1 (post-install dep check)
- **Issue:** RESEARCH predicted `recharts@^3.8.1` based on PR #8486 having merged into shadcn's chart registry, but the shadcn CLI v4 still resolves to `recharts@2.15.4`. Initial manual install of v3 was then overwritten by the CLI's pinned v2.
- **Fix:** Accepted registry authority. Public API Sparkline uses (Area, AreaChart, ChartContainer wrapper) is identical across v2 → v3 — no migration cost. Recorded in decisions/metrics.
- **File modified:** `package.json` (recharts `^2.15.4`)
- **Commit:** `1e4b2fb`

**6. [Rule 3 — Workspace] npm install cleanup hit Windows EPERM on first shadcn CLI run**
- **Found during:** Task 1 pre-flight npm install
- **Issue:** First `npx shadcn add chart slider avatar` attempt failed mid-install with `ENOTEMPTY: directory not empty, rmdir '...\node_modules\next\dist\server\lib\router-utils'` — Windows file-lock on a running Node process's temp dirs.
- **Fix:** Ran a manual `npm install @radix-ui/react-slider @radix-ui/react-avatar recharts` to seat the deps, then re-ran shadcn CLI with `--overwrite`. Second run completed cleanly. Root cause is environmental (Windows worktree file locks), not a code bug.
- **No permanent file change required.**
