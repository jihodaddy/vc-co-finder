## Summary

Phase 03.1 Waves 1–6 deliver the brand design system migration (cream / ink /
lime palette, Pretendard + Geist fonts, metrics primitives, facet-chip Badge
variant, dual-thumb Slider, Wave-6 freshness dot recoloring, minimal hero
landing). The implementation is cohesive, tokens are centralized in
`globals.css`, and tests cover the important invariants (token hex values,
filter-chip rendering, Slider dual-thumb, freshness hex guard, neutral-scrub
audit).

No Critical issues. The three Warnings are:

1. **i18n shape drift (`en.json` missing `search.chip.range.*` subtree)** — not
   just empty strings. At runtime under `en` locale, `ActiveFilterChips` will
   throw `MISSING_MESSAGE` for every funding/founded/employees-custom range
   filter. `search.results.liveLabel` is also absent from `en.json` and used
   by `ResultsHeader`.
2. **`Sparkline` uses a hardcoded `id="spark-fill"`** on its `<defs>
   <linearGradient>`. Multiple instances on the same page emit duplicate DOM
   ids, and every `fill="url(#spark-fill)"` resolves to the first gradient —
   which becomes visually wrong once the consumer passes different `color`
   props to sibling sparklines. Phase 6 will light this up.
3. **`FacetRangeInputs` uses `Number(bigint)` to coerce funding amounts to
   slider positions**. For amounts above `Number.MAX_SAFE_INTEGER` (~9 PB KRW
   — technically unreachable with the current 1조 domain cap, but the
   `parseKRW` return type is `bigint` for exactly this reason) the conversion
   loses precision. Today harmless, but worth a small guard rail before the
   cap is ever raised.

Info items cover minor a11y polish, dead i18n keys, a sparkline data
contract gap, and a styling convention note. No source files were modified.

## Warnings

### WR-01: `en.json` missing `search.chip.range.*` subtree (and `search.results.liveLabel`) — runtime crash under `en` locale

**File:** `src/messages/en.json:190-194`, cross-referenced at
`src/components/search/ActiveFilterChips.tsx:109-185` and
`src/components/search/ResultsHeader.tsx:28`

**Issue:** `ko.json` defines
`search.chip.range.{between,atLeast,atMost,foundedAfter,foundedBefore}` (ko
lines 194-200), but in `en.json` (line 190) the `chip` block contains only
`remove`, `clearAll`, `activeLabel` — the `range` sub-object is absent
entirely. Likewise `search.results.liveLabel` is used by `ResultsHeader.tsx`
line 28 but `en.json` defines only `liveRegion` (line 197). `routing.ts`
lists `en` as a fully supported locale (`locales: ['ko', 'en']`), so a user
hitting `/en/search` with any funding/founded/employees-custom-range filter
active will trigger a `MISSING_MESSAGE` runtime error from next-intl in
`ActiveFilterChips`, and `ResultsHeader` will throw on every search page.

Per-project policy `en.json` is intentionally empty-stringed for v1 — that
IS acceptable and does not throw (empty-string messages render as empty
nodes). But a missing key is a different failure mode: next-intl throws by
default.

**Fix:** Add the missing leaves to `en.json` as empty strings, matching the
pattern used elsewhere. Minimum required patch:

```jsonc
// src/messages/en.json — inside "search": { ... }
"chip": {
  "remove": "",
  "clearAll": "",
  "activeLabel": "",
  "range": {
    "between": "",
    "atLeast": "",
    "atMost": "",
    "foundedAfter": "",
    "foundedBefore": ""
  }
},
"results": {
  "count": "",
  "liveLabel": "",       // <-- add
  "liveRegion": "",
  "sortBy": "",
  "view": { ... }
},
```

Alternative (broader fix): add an `onError` handler to
`NextIntlClientProvider` / `getRequestConfig` that downgrades
`MISSING_MESSAGE` to an empty string for `en` only — but parity patch is
simpler and keeps the v1 KR-only policy loud in the file.

### WR-02: `Sparkline` uses a hardcoded SVG `id="spark-fill"` — breaks multi-instance rendering

**File:** `src/components/metrics/Sparkline.tsx:54-64`

**Issue:**

```tsx
<defs>
  <linearGradient id="spark-fill" x1="0" y1="0" x2="0" y2="1">
    <stop offset="0%" stopColor="var(--color-v)" stopOpacity="0.25" />
    <stop offset="100%" stopColor="var(--color-v)" stopOpacity="0" />
  </linearGradient>
</defs>
<Area ... fill="url(#spark-fill)" />
```

The `id` is a literal string. Per the HTML spec every element id must be
unique in a document. As soon as two `<Sparkline>` instances render on the
same page (Phase 6 will absolutely do this — a list row per company, or a
profile page with several metric sparklines), two `<linearGradient>` nodes
will share the id, and every `<Area fill="url(#spark-fill)">` — regardless
of which `<Sparkline>` it's inside — will resolve to the **first** gradient
in the DOM. If the consumer passes different `color` props to siblings, all
sparklines render with the first one's color. It's also an a11y / validity
bug even when colors happen to match (duplicate ids).

**Fix:** Use `React.useId()` to scope the gradient per instance. Recharts
instances already use this pattern in their docs:

```tsx
import { useId } from 'react';
// ...
export function Sparkline({ data, color = 'var(--color-chart-1)', width = 120, height = 36, className }: SparklineProps) {
  const gradientId = useId().replace(/:/g, '');
  const fillUrl = `url(#${gradientId})`;
  // ...
  return (
    <ChartContainer ...>
      <AreaChart ...>
        <defs>
          <linearGradient id={gradientId} x1="0" y1="0" x2="0" y2="1">
            <stop offset="0%" stopColor="var(--color-v)" stopOpacity="0.25" />
            <stop offset="100%" stopColor="var(--color-v)" stopOpacity="0" />
          </linearGradient>
        </defs>
        <Area ... fill={fillUrl} />
      </AreaChart>
    </ChartContainer>
  );
}
```

(`ChartContainer` already uses `React.useId()` at `chart.tsx:46` — same
pattern.) Optional follow-up: add a vitest case that renders two
`<Sparkline>` with different colors and asserts both gradient `id`s are
unique.

### WR-03: `FacetRangeInputs` narrows `bigint` → `Number` on the funding slider path — silent precision loss past 2^53

**File:** `src/components/search/FacetRangeInputs.tsx:61-87`

**Issue:** `sliderMin` / `sliderMax` do:

```tsx
const p = parseKRW(minTxt);      // returns bigint | null
if (p === null) return domainMin;
const n = Number(p);             // <-- narrows bigint → number
return Math.max(Math.min(n, domainMax), domainMin);
```

`parseKRW` returns `bigint` precisely to preserve precision over
`Number.MAX_SAFE_INTEGER` (9.007 × 10¹⁵). Today `MAX_FUNDING = 1 × 10¹²`
(1조) so the narrowing is currently safe (1 trillion < MAX_SAFE_INTEGER).
However:

1. If `MAX_FUNDING` is ever raised to 1경 (1 × 10¹⁶) or user input exceeds
   2^53 before being clamped by `Math.min(n, domainMax)`, the
   `Number(bigint)` step will silently lose precision **before** the clamp
   — so the clamp can produce a wrong value.
2. The pattern sets a precedent that future contributors may copy without
   the invariant.

No runtime defect today; this is a "guard-rail before cap is raised" note.

**Fix:** Clamp in bigint space before narrowing, or use `BigInt(domainMax)`:

```tsx
const maxBig = BigInt(domainMax);
const minBig = BigInt(domainMin);
const clamped = p > maxBig ? maxBig : p < minBig ? minBig : p;
return Number(clamped); // safe — always ≤ domainMax which fits in Number
```

This makes the safe-to-narrow invariant explicit in code, not implicit in
`MAX_FUNDING`'s current value. The `founded` branch (plain-integer years)
is unaffected.

### IN-02: `Sparkline` accepts `data: number[]` with zero guard for empty / single-point series

**File:** `src/components/metrics/Sparkline.tsx:31-68`

**Issue:** `Sparkline({ data: [] })` will render an empty `<AreaChart>`;
`Sparkline({ data: [42] })` draws a flat single-point area that Recharts
treats as a degenerate path. Neither crashes, but the visual result is
indistinguishable from "chart broken" — the primitive silently accepts
bad input rather than signaling "no data" vs "one point" vs "trend".

Phase 6 will be the consumer so it's acceptable to leave the guard to that
phase, but worth documenting in `SparklineProps`:

```tsx
/**
 * Minimum 2 points. Callers with < 2 points should render a placeholder
 * (e.g. a dash "—") instead of invoking Sparkline.
 */
data: number[];
```

**Fix:** Add the JSDoc note above, or add a runtime guard:

```tsx
if (data.length < 2) return null;
```

Returning null is safer than rendering a confusing shape. Phase 6 can
override if the product wants a "no data" visual.

### IN-03: `Delta` primitive hardcodes hex colors (documented deviation)

**File:** `src/components/metrics/Delta.tsx:20-22`

**Issue:** `#0C8A3A` and `#C03A3A` are inlined rather than routed through
`--chart-*` tokens. The file comment explicitly flags this as a
"primitive-internal value" decision (DESIGN.md §Delta), and the same hex
is reused by `FRESHNESS_DOT_CLASS` at `src/lib/data/freshness.ts:51-53`
for fresh/expired. Since the same two hex strings now appear in two
places (`Delta.tsx` and `freshness.ts`) with different rationales, a
single-source-of-truth move would be safer.

**Fix (optional):** Promote these to CSS variables in `globals.css`:

```css
--color-signal-up:   #0C8A3A;  /* Delta ▲ / freshness fresh */
--color-signal-down: #C03A3A;  /* Delta ▼ / freshness expired */
```

Then `Delta.tsx` references `var(--color-signal-up)` and `freshness.ts`
uses the same via Tailwind arbitrary-value syntax. Zero behavioral change;
one place to update on brand tweaks. Current tests still guard the exact
hex so the migration is cheap.

### IN-04: Badge `dismissAriaLabel` fallback to empty string weakens screen-reader contract

**File:** `src/components/ui/badge.tsx:45-48, 74`

**Issue:** The JSDoc says "consumers MUST pass an i18n-localized string"
but the prop is optional and falls back to `""` — a dismiss button with
`aria-label=""` is effectively unnamed for screen readers and violates
WCAG 2.1 SC 4.1.2 (Name, Role, Value). Current call sites do pass a
label (`ActiveFilterChips.tsx:223`), but the type doesn't enforce it.

**Fix:** Either make the prop required when `dismissible` is true via a
discriminated union, or add a `process.env.NODE_ENV !== 'production'`
dev-time warning:

```tsx
// Option A: require via type
type DismissibleProps =
  | { dismissible?: false; onDismiss?: never; dismissAriaLabel?: never }
  | { dismissible: true; onDismiss: () => void; dismissAriaLabel: string };

// Option B: dev warning
if (process.env.NODE_ENV !== 'production' && dismissible && !dismissAriaLabel) {
  console.warn('Badge[dismissible]: dismissAriaLabel is required for a11y');
}
```

Option A is stricter and catches issues at compile time.

### IN-05: `FacetRangeInputs` slider `min === max === 0` edge case when user clears both inputs with an empty query param

**File:** `src/components/search/FacetRangeInputs.tsx:131-144`

**Issue:** When `query.funding === ''` and the user has not typed anything,
`sliderMin` returns `domainMin = 0` and `sliderMax` returns
`domainMax = 1_000_000_000_000`. Good. But if a URL arrives with
`?funding=-` (trailing dash — e.g., from an external malformed share),
`parseRange` yields `{min: '', max: ''}` so the same defaults apply. That's
actually fine, but there's a subtler path: if a user types "1000" in min,
then clears max entirely, `minTxt.trim()` is non-empty and `maxTxt.trim()`
is empty → `sliderMin ≈ 1000`, `sliderMax = domainMax`, `min > max` never
triggers (good). No defect, just noting that the slider thumbs drift
to the domain edges when inputs are partially filled, which may visually
surprise users.

**Fix:** Low priority. Consider a UI note in the range helper / aria
description that blank max = "no upper bound". Not a blocker.
